Herhaling — Module 3
Relaties, ERD'ish en kardinaliteit
▼

We starten met een snelle herhaling van module 3. Klik de kaartjes om de begrippen op te frissen!

↔️1:1 relatieweet jij het nog?
Elk record in A hoort bij precies één record in B en andersom. Bijv. employee ↔ car.
🌿1:N relatieweet jij het nog?
Één record in A hoort bij meerdere records in B. Bijv. house → resident. FK staat aan de N-kant.
🔀M:N relatieweet jij het nog?
Meerdere records in A horen bij meerdere records in B. Moet worden opgelost met een tussentabel.
✍️ERD'ishweet jij het nog?
"Each Entity1 may/must verb 1 and only 1 / 1 or more Entity2." — Standaardzin om relaties te beschrijven.

Wat is een M:N relatie?
Veel-op-veel — alles hangt met alles samen
▼

Je hebt al geleerd hoe 1:1 en 1:N relaties werken. Nu de derde — en lastste — categorie: de veel-op-veel-relatie (M:N).

📌 Definitie M:N relatie
Een veel-op-veel-relatie betekent dat meerdere records van entiteit A gekoppeld kunnen zijn aan meerdere records van entiteit B — en andersom.

💻 Voorbeeld: computer ↔ software

Een computer kan meerdere softwareprogramma's hebben. Dezelfde software kan op meerdere computers staan. Dat is M:N.

// Welke software staat op welke computer?
🖥️ Computer 1 🖥️ Computer 2 🖥️ Computer 3
📦 Word ✓ ✓ —
📦 Chrome ✓ ✓ ✓
📦 Photoshop — — ✓

Computer 1 heeft Word én Chrome. Chrome staat op álle drie computers. Dit is een M:N relatie.

De ERD'ish zinnen herkennen als M:N

Each computer may have 1 or more software. 1 or more →
Each software may be on 1 or more computer. 1 or more →
🚨 Signaal: twee keer "1 or more" = M:N!
Als beide ERD'ish zinnen eindigen op "1 or more", heb je een veel-op-veel-relatie. Dit kan niet direct in een database worden opgeslagen — je moet het oplossen met een tussentabel.

Welk van de volgende scenario's is een M:N relatie?

📚 Een student kan meerdere cursussen volgen. Een cursus kan meerdere studenten hebben.

🚗 Een werknemer heeft maximaal één bedrijfswagen.

Waarom moet je M:N oplossen?
Drie problemen zonder tussentabel
▼

Je kunt een M:N relatie proberen op te slaan in één tabel — maar dat geeft drie grote problemen. Ontdek ze stap voor stap.

❌ Probleem 1 — Inconsistentie & fouten

Typefouten en inconsistente namen

Als je namen gebruikt in plaats van ID's, kan dezelfde software op twee manieren gespeld worden:

computernamesoftwarename
Computer1Microsoft Word
Computer2MS Word ← ⚠️ typefout!
Computer3Microsoft Word

De database ziet "Microsoft Word" en "MS Word" als twee verschillende programma's — terwijl het hetzelfde is.

❌ Probleem 2 — Beheerlast bij wijzigingen

Naam wijzigen = overal aanpassen

Stel: "Computer1" wordt hernoemd naar "Werkplek-A". Dan moet je elke rij in de tabel handmatig aanpassen:

computernamesoftwarename
Computer1Microsoft Word
Computer1Google Chrome
Computer1Photoshop

Met ID's hoef je maar één rij te wijzigen in de computer tabel.

❌ Probleem 3 — Extra opslag

Volledige namen kosten veel ruimte
❌ Met namen (inefficiënt)
"Computer1" · "Microsoft Word"
"Computer1" · "Google Chrome"
"Computer2" · "Microsoft Word"
"Computer2" · "Google Chrome"
→ ~40 bytes per rij
✅ Met ID's (efficiënt)
1 · 101
1 · 102
2 · 101
2 · 102
→ ~4 bytes per rij
✅ Conclusie — Gebruik altijd een tussentabel
Een tussentabel (ook wel intersection entity of koppeltabel genoemd) lost alle drie de problemen op:
  • Geen typefouten — namen staan maar op één plek
  • Naam wijzigen = maar één rij aanpassen
  • Opslag met ID's is compact en snel

De tussentabel
Hoe los je een M:N relatie op?
▼

Een M:N relatie los je op in 4 stappen. Na deze stappen heb je twee 1:N relaties in plaats van één M:N.

📋 Stap 1 — Identificeer de twee entiteiten

Stel vast dat er een M:N relatie bestaat door de ERD'ish zinnen te schrijven. Als beide zinnen eindigen op "1 or more" → M:N!

Each computer may have 1 or more software.
Each software may be on 1 or more computer.

→ Twee keer "1 or more" = M:N relatie ✓

📋 Stap 2 — Maak een tussentabel

Voeg een nieuwe entiteit toe tussen de twee. Naam: combineer de twee entiteiten of beschrijf de relatie logisch.

computer
  • 🔑 id
  • ◆ computername
→
softwarerow ✨
  • 🔑 id
  • 🔷 computer_id
  • 🔷 software_id
  • ◇ installdate
←
software
  • 🔑 id
  • ◆ softwarename

De tussentabel softwarerow bevat de FK's van beide entiteiten en optionele extra attributen.

📋 Stap 3 — Schrijf de nieuwe ERD'ish zinnen

De M:N is nu opgesplitst in twee 1:N relaties:

Each computer may have 1 or more softwarerow.
Each softwarerow must belong to 1 and only 1 computer.
Each software may have 1 or more softwarerow.
Each softwarerow must belong to 1 and only 1 software.
📋 Stap 4 — Het opgeloste ERD

Het resultaat: twee 1:N relaties met de tussentabel in het midden.

1 / 4

🛒 Nog een voorbeeld: order ↔ product

Denk aan een supermarktbonnetje: één order heeft meerdere producten, één product kan in meerdere orders voorkomen.

❓ Wat bevat de tussentabel orderrow minimaal?
De namen van order en product
De primaire sleutels van order en product als FK's
Alleen een eigen id
Alle attributen van beide entiteiten

Alternatieve primaire sleutel
Waarom een eigen id in de tussentabel?
▼

Een tussentabel heeft altijd minstens twee FK's. Maar je kunt ook een eigen primaire sleutel (id) toevoegen. Dit noem je een alternatieve primaire sleutel.

Zonder vs met alternatieve PK

Zonder alternatieve PK
computer_id (PK)software_id (PK)installdate
11012024-06-01
11022024-06-03
21012024-06-05

PK = samengesteld uit twee kolommen. Moeilijker te gebruiken als andere tabellen naar deze tabel willen verwijzen.

✅ Met alternatieve PK
id (PK)computer_id (FK)software_id (FK)installdate
111012024-06-01
211022024-06-03
321012024-06-05

PK = enkelvoudig id. Eenvoudig te verwijzen vanuit andere tabellen, makkelijker te querien.

💡 Stappen: alternatieve PK toevoegen
  1. Voeg attribuut id toe aan de tussentabel
  2. Stel id in als primaire sleutel (PK)
  3. Maak van computer_id en software_id vreemde sleutels (FK)
